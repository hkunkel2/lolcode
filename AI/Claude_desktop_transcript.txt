Here's the practical guidance:
Where you should use ///:

All public items: functions, methods, structs, enums, traits, type aliases, constants, and modules. This is the strong community convention and #[warn(missing_docs)] can enforce it.
Trait methods — both the trait definition and any non-obvious impl.

Where you can skip them:

Private items — nice to have, but not expected.
Self-explanatory one-liners like fn is_empty(&self) -> bool on a collection (though even these often get a one-liner doc in the standard library).
Fields of structs, unless their purpose isn't obvious from the name.

Best practices:

First line should be a short summary sentence (this becomes the synopsis in generated docs).
Use # Examples, # Panics, # Errors, and # Safety sections where relevant.
Include a runnable example with ```rust blocks — cargo test runs these as doctests, so they double as tests.
Use //! (inner doc comments) at the top of lib.rs or mod.rs to document the crate/module itself.

The general rule: if it's pub, document it with ///. If it's private, use regular // comments where helpful.
